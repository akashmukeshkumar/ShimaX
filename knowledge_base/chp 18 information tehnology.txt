🛡️ Risk Assessment & Controls
Jab risk assessment start hota hai to auditor ko internal controls samajhne zaroori hote hain.
Controls do types ke ho sakte hain:
Manual Controls – Example: Payment ke liye invoice pe “Paid” ka stamp lagana.
Automated Controls – Example: Vendor Management System (VMS).
📝 Example of Manual Control
Company me rule hai: Koi bhi payment invoice/supporting document ke bina nahi hogi.
Salary → Salary sheet required.
Purchases → Invoice required.
Invoice pe Paid ka stamp lagana zaroori hai taake duplicate payment na ho.
💻 Example of Automated Control (PTCL – Vendor Management System)
Auditor ne PTCL se poocha: “How do you ensure ke double payment na ho?”
PTCL ne bataya: hum use karte hain Vendor Management System (VMS).
Process:
Purchase Order (PO) raised – jab koi cheez/service kharidni hoti hai.
Vendor ko approve list me add kiya jata hai.
Vendor apni invoice system me upload karta hai (human intervention remove ho jati hai).
System existence check lagata hai:
Invoice → PO se match hoti hai ✅
GRN (Goods Received Note) se verify hoti hai ✅
Agar sab match ho jaye to system payment allow karega.
Agar mismatch ho jaye → Payment block ho jati hai.
🔄 Flow of Transactions (PTCL Case)
Goods receive hote hi system me GRN (Goods Received Note) generate hota hai.
Accounting entry:
Dr. PPE (Asset)
Cr. GR/IR (Goods Received/Invoice Received) Liability
Jab actual invoice upload hoti hai aur PO se match hoti hai:
GR/IR liability convert hoti hai actual liability me.
Tabhi Bank Payment Voucher generate hota hai.
CFO approval deta hai aur vendor ko payment release hoti hai.
🧩 Separation of Duties
Fraud avoid karne ke liye different departments responsible hote hain:
Purchase Dept → PO banata hai.
Stores/Inspection → Goods receive karke GRN banata hai.
Finance Dept → Payment settlement karta hai.
Is separation se fraud aur misstatement ka risk reduce hota hai.
🔄 Automated Process Flow
Auditor ne explain kiya ke system itna automatic design hota hai ke bina proper liability record kiye Bank Payment Voucher (BPV) generate hi nahi hota.
Example: Invoice upload → match with Purchase Order (PO) ✅ → debit vendor table → phir hi BPV generate hoga.
Agar liability exist nahi karti → system error throw karega (on-screen error message ❌).
🧾 Invoice & PO Matching
Invoice system me upload hoti hai → PO ke sath auto-match hoti hai.
Agar vendor ne duplicate invoice dal di:
System bolega: “Invoice number already exists” ⚠️
Agar invoice amount > PO amount → system limit check karega 🚫
Matlab system har jagah control laga raha hai:
Existence Check
Limit Check
👨‍💼 CFO ka Role
CFO ke paas BPV tabhi jayega jab sari checks clear ho chuki hongi ✔️
Isliye CFO tension-free hota hai 😎
🔐 Types of Controls in Process
Existence Check → Duplicate invoice ya entry rokna.
Limit Check → PO ke upar amount allow nahi hota.
Matching Controls → PO, GRN (Goods Receipt Note) aur Invoice teeno ka match hona.
Error Messages → Agar koi issue hai to system turant block kar deta hai.
🏭 Real-World Example: PTCL SAP System

🤝 Manual vs IT Controls
Manual controls abhi bhi exist karte hain → jaise ek banda GRN aur liability reconcile kare.
Lekin automated controls fraud detection aur prevention zyada strong bana dete hain.
🔗 Fraud Linkage
Fraud risk: “Payment has been made for goods not received” 🚚💸
Automated controls prevent karte hain:
Invoice bina GRN ke process nahi hoti.
Related party transactions bhi isi system se pass hoti hain → sirf board approval ka extra step hota hai 📝.
🔹 Controls ka concept:
Agar process clear hoga tabhi control effective hoga ✅
Example: PTCL aur Vendor case.
Manual control: Invoice chipka dete thay — weak control.
Automated control: System allow hi nahi karta jab tak PO match na ho 🖥️.
🔹 Limit Check Control 💰:
Payment process me lagta hai.
Agar certain amount se upar ho jaye → system allow hi nahi karta 🚫.
Preventive control: Pehle hi rok deta hai.
Exception Report: Jab allow kar deta hai aur baad me report generate hoti hai 📑 (detective control).
🔹 Invoice Upload Process 🧾:
Invoice sirf management system portal pe upload hoti hai.
System automatically PO aur Invoice match karta hai ✅.
Agar mismatch ho jaye → upload hi nahi hoti ❌.
🔹 Preventive vs Detective Controls ⚖️:
Preventive: System upfront rok deta hai (e.g., amount > limit).
Detective: Exception report generate hoti hai → baad me rectify karte hain.
🔹 PTCL Case Study 📊:

Invoice me higher amount dala → system ne reject kar diya ✅.
PO aur Invoice ka strict matching → Collusion avoid hoti hai.

🔹 Limit Check ka aur ek Example:

🔑 Key Concept: Data Transmission & Encryption Controls
Data Transmission Control: Jab bhi sensitive data share hota hai, ensure karo ke confidential rahe aur unauthorized access na ho.
Methods of securing data:
Password-protected files: Basic level of protection.
Encryption: Convert data into unreadable format (1010 binary or coded language) → Only intended recipient can decrypt.
🔐 Types of Encryption
Symmetric Encryption:
Single key/password for encryption & decryption.
Example: Data send karo → Recipient ke paas same password → Data decrypt kare.
Asymmetric Encryption:
Alag keys for encryption & decryption.
Sender encrypts with public key → Recipient decrypts with private key.
Application: Different files can use different methods (some symmetric, some asymmetric) depending on sensitivity.
🛡️ Data Transmission Scenarios
Confidential financial data: e.g., payroll, bank transfers.
Secure transfer requirement: Ensure data only reaches intended recipient.
Authentication codes: Required to access data. Without code, recipient cannot read/receive data.
Example: Mobile-based transaction code for rural areas (EasyPaisa, wallet). Sender provides code → Recipient verifies → Transaction completes.
🖥️ Controls for Sensitive Data
Access Control: Only authorized personnel can access data.
Acknowledgement Code: Recipient must confirm receiving/reading data.
Audit Trail: Sender can verify if recipient opened/read the email.
Application in Payroll:
Employee removal from payroll after leaving organization.
Removal triggers an acknowledgement code for verification.
Notification Systems: Inform sender when recipient reads the message/email.
💻 Real-life Examples & Applications
Email: Important confidential emails sent with encryption & acknowledgement code.
Mobile Transaction (EasyPaisa): Transaction cannot proceed without entering authentication code.
Tracking access: Sender can confirm if recipient opened/read the information.
Integration with ID control: Authentication code acts as input authorization control.
Topic: Data Transmission, Network Security, Program Control, Micro vs Inline Systems
1️⃣ Data Security & Control Measures
Network Security:
Apne network ko secure rakhna zaroori hai, warna kisi ke paas password hoga to hacking ho sakti hai.
Wi-Fi ya public network se connect hone se risk zyada hai.
Personal devices ko organizational network se connect na karein bina authorization.
Authentication & Encryption:
Data ko encrypt karke aur authenticated codes ke saath transfer karein.
Program control ensures data format consistency.
Real-time monitoring aur update check zaroori hai.
Program Control:
Data transfer ka program control lagana zaruri hai.
Data ka format check hota hai aur sender & receiver ke versions match hone chahiye.
Authentication codes use kar ke ensure kiya jata hai ki data correct aur secure hai.
2️⃣ Payroll & Transaction Example
Example of controlling sensitive payroll data:
Employee resigns → removal from department & payroll.
Acknowledgment codes ensure that the receiving person has actually received and read the information.
Helps prevent errors and unauthorized access.
Lesson:
Controlled data transmission ensures no financial loss and proper tracking.
3️⃣ Micro Systems vs Inline (Integrated) Systems
Micro (Non-Integrated) Systems:
Not centralized; branch-wise independent.
Example: State Life Insurance – each branch had separate ledgers; consolidation at head office.
Issues: version mismatch, incomplete data, more chances of error.
Inline / Integrated Systems:
Centralized & real-time.
All branches work on same version; data immediately reflects in head office.
Pros: accurate, consistent, timely.
Cons: expensive, requires high-end training.
Real-Life Example:
Bank loans and income book: prior suspension → reversal per new regulation.
Micro system: reconciliation issues, delays.
Integrated system: real-time, fewer errors.
4️⃣ Password & Logical Security
Password change reminder: every 15 days to prevent unauthorized access.
Logical security ensures that confidential data is safe even if someone sees the physical environment.
Automated notifications to users for compliance.
